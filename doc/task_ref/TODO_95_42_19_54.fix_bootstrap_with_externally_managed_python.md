Now I have the full picture. Here's the situation:

  The bootstrap chicken-and-egg problem:
  - uv is not in PATH — it only lives inside cache/venv/uv.venv/bin/uv (which protoprimer creates)
  - To create uv.venv, VenvDriverPip._create_venv_impl() (line 786) calls python3.14 -m venv, which triggers ensurepip, which fails because the EXTERNALLY-MANAGED marker exists at
  sys.prefix/lib/python3.14/EXTERNALLY-MANAGED

  Detection is straightforward:
  import sys, os
  marker = os.path.join(sys.prefix, 'lib', f'python{sys.version_info.major}.{sys.version_info.minor}', 'EXTERNALLY-MANAGED')
  is_externally_managed = os.path.exists(marker)

  What to do when detected — three realistic options:

  1. --without-pip + bootstrap pip separately: Create venv with python3.14 -m venv --without-pip uv.venv, then run ensurepip using a different, non-externally-managed Python (e.g.
  /usr/bin/python3) to seed pip into it. Fragile — depends on system Python availability.
  2. Fall back to system Python for the intermediate uv.venv: The uv.venv only needs uv installed in it — it doesn't need to match the user's required Python version. If the selected
  Python is EXTERNALLY-MANAGED, use sys.executable of the current running bootstrap Python (which is presumably a plain system Python) to create uv.venv instead.
  3. Use uv's own installer to bootstrap: Download and run https://astral.sh/uv/install.sh (or the equivalent pip-installable uv wheel from PyPI via --break-system-packages). This makes uv
  available system-wide, bypassing the whole uv.venv approach. But it modifies the user's environment outside protoprimer's cache.

  Option 2 is the cleanest fit for protoprimer's architecture: _ensure_uv_is_available() could detect the EXTERNALLY-MANAGED marker on selected_python_file_abs_path, and if present,
  substitute the running interpreter (sys.executable) as the Python used to create the intermediate uv.venv. The final main venv still uses the required Python version via uv.

Yes, you're correct. selected_python_file_abs_path is resolved by shutil.which("python3.14") (line 5777) which finds /home/uvsmtid/.local/bin/python3.14 because ~/.local/bin is first in
  PATH. That's a PATH lookup, not sys.executable — but the result is the same problematic Python.

  And if protoprimer itself is being run by that same python3.14, then sys.executable also points there. So option 2 as I described it — "fall back to sys.executable" — fails for the exact
  same reason.

  To make option 2 actually work, the fallback Python search would need to explicitly skip candidates with the EXTERNALLY-MANAGED marker. Something like: walk through PATH candidates for
  the required version, check <sys.prefix>/lib/pythonX.Y/EXTERNALLY-MANAGED, and skip any that have it. That would eventually land on /usr/bin/python3.12 (or whatever system Python doesn't
  have the marker) to create the intermediate uv.venv.

  But that introduces a new problem: the fallback Python may not match the required version — you'd be creating uv.venv with 3.12 just to install uv, when the project requires 3.14. That's
  actually fine since uv.venv is only an intermediate vehicle to get uv installed, not the main venv. But it's a non-obvious constraint worth documenting.
