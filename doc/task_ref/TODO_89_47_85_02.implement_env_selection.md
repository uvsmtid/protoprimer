## Background

At the moment, the required environment is not selected automatically.

So, if you call bootstrap twice:

*   with `--env` arg:

    it bootstraps with the required environment config

*   without `--env` arg:

    it fails because the default environment config is selected conflicting with the current symlink

Figure out how to deal with this situation.

## Approach A: follow the "single-touch" phylosophy and avoid `--env` arg

This way re-evaluation of the selected environment should be the same, and bootstrap will work
(both cases will avoid `--env` but the required environment is chosen automatically).

## Approach B: always respect the current symlink as the selected environment.

This way there will never be conflict in the second call.
The bootstrap process will simply re-select the same environment based on the existing symlink.

## Approach C: implement both A and B.

It seems like both A and B make sense.

Moreover, it still allows manual selection of the environments via `--env`.
