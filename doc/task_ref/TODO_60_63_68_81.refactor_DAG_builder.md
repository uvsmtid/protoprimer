## Refactoring goals

The objective is to refactor the `DAG_builder` to build
sub-graphs (also DAG-s) based on the current `GraphCoordinates`.

This eliminates complex `if/else` logic within `EnvState`-s to avoid `python` restarts for specialized
modes like `start_app` or `get_derived_config`.

For example:
*   calling `start_app` does not require most of `venv` creation and population steps
*   calling `get_derived_config` must happen with no `python` restarts
*   both `start_app` and `get_derived_config` should not process any CLI args (as `env_boostrapper` does)

## Implementation decisions

*   DAG sub-graphs

    Maintain a single `EnvState` list representing the "maximum" potential DAG.

    Specialized modes will simply use reduced sub-sets of this list.

*   DAG addressing

    Use a 1-dimensional list of names (`EnvState` enum) to address DAG nodes.

    But an N-dimensional space (one of the enums listed in `GraphCoordinates`) to address DAG sub-graphs.

    In other words:
    *   `EnvState` enum items are 1-dimentional space.
    *   `Bootstrap_*` classes live in N-dimensional space (mapped from 1-dimentional to N-dimentional by factories).

*   Factory mapping

    Each `EnvState` maps into a factory.

    The factory instantiates one of the `Bootstrap_*` classes (based on `GraphCoordinates`).

    The `Bootstrap_*` class then identifies its own dependencies (`EnvState` enums)
    to be instantiated via their own factories.

*   Dynamic DAG

    The full DAG is never constructed at once (statically).

    Instead, DAG is built backwards from a specific target `EnvState`.

*   Multiple targets

    Since `GraphCoordinates` are unknown until discovered (through environment variables or CLI args),
    the DAG set new targets during execution.

    For example, the role of `proto_main` shifts to:

    *   pulling an initial target `EnvState.state_sub_command_executed`
    *   executing that partial DAG: `EnvState.state_sub_command_executed` re-pulls the rest via `RunStrategy`
    *   using the resulting context to determine the next target

*   `GraphCoordinates`

    Separate:

    *   `EntryFunc`: known immediately via the entry point:
        *   `boot_env` (CLI processing)
        *   `start_app` (no CLI processing)
        *   API library (no CLI processing, no `python` restarts)
        *   direct `proto_main` execution (similar to `env_boostrapper`)
    *   `SubCommand`: known only after CLI argument processing

*   `EnvState`-s

    State names should remain stable and known at the start of a `StateStride`, though they can be extended in an append-only fashion.
