## Specific problem

We want to simplify logic and DAG for some of the `RunMode`-s.

For example:
*   calling `app_starter` does not require most of `venv` creation and population steps
*   calling `get_derived_config` must happen with no `python` restarts

Another example:
*   both `app_starter` and `get_derived_config` should not process any args as if it is `protoprimer`

At the moment, this is done by invoking the only implementation of `EnvState`
which does `if`-ing/`else`-ing to skip execution and unnecessary `python` restarts.

If it was a different simplified specialized DAG implementation, it would be easier to follow. And that is the goal.

## Confirmations

*   How do we address DAG nodes?

    We can have some discrete N-dimentional space (N enum classes) where each mode resides, but it is an overkill for the single script.

    The simplest is a flat 1-dimentional list of names.

*   Do we want to use a single `EnvState` list?

    Yes. See addressing DAG nodes. It represents the max DAG for `protoprimer` (without custom extensions) which does full bootstrap - everything else is reduced from there.

*   Do we want to use single set of `Bootstrap_*` implementations?

    No/Yes.

    No because the purpose is to simplify implementation of individual steps and remove if-ing/else-ing based on modes. The sub-set to execute will be different depending on the mode.

    Yes because the full set of implementations is still single, it is just that not every `Bootstrap_*` will be plugged in in every run.

*   How `EnvState` names and `Bootstrap_*` implementations relate?

    Each `EnvState` is mapped into a factory. We plug specific `Bootstrap_*` implementation while calling the factory of each mode. Factory provides a `Bootstrap_*` instance which, in turn, provides names of the nodes it depends on (this will lead to factories of dependent nodes).

*   When do we construct full DAG?

    We never construct full DAG.

    We only construct partial DAG backwards from the target. And we may set multiple different targets while executing progressively.

    For example, the full DAG depends on `RunMode`,
    but `RunMode` is only known after some partial DAG execution (to load initial values from env vars and CLI args).
    We cannot create that full DAG in advance (unless we want to deal with universal complex unspecialized DAG contradicting the purpose of this refactoring).

*   What is the role of `protoprimer.primer_kernel.app_main`?

    The role of main will be to pull initial target. That target will cause creation of some DAG, execute it, and populate the context to pull next target.

    For example, at the moment, `app_main` constructs full DAG, but the execution is twisted:
    *   `app_main` only pulls `EnvState.state_run_mode_executed`
    *   `EnvState.state_run_mode_executed` re-pulls the rest via `RunStrategy`
    *   `RunStrategy` (mostly `ExitCodeReporter`) invokes `sys.exit` with the exit code from state.

*   What is the role of `protoprimer.primer_kernel.RunStrategy`?

    At the moment, there is single strategy `ExitCodeReporter` (see `EnvState.state_run_mode_executed`).

    Do we want to make use of `RunStrategy` beyond that degenerate single case?

## Other refactoring

*   Split `RunMode` and `StartMode`.

    *   `RunMode` is only known after CLI args are processed.
    *   `StartMode` is known immediately via API used: env_bootstrapper, app_starter, CLI, lib, ...

    Use `StartMode` for initial (partial) DAG execution. Then, depending on `StartMode`, it is going to evaluate `RunMode` or not.

*   DAG options:

    *   Always full DAG?

        The entire DAG is built and known immediately (static) on start
        (by building `EnvContext` whether using default `app_main` or not).

        Execution is controlled by enum values - you can only execute those parts when those enums match.

        Pros:
        *   Simpler than dynamic DAG.
        Cons:
        *   No dynamic dependencies (some states will have to have noop `Bootstrap_*` implementations.

    *   Extensible DAG?

        The full DAG is unknown, it grows depending on the previous state execution.

        Pros:
        *   Very flexible.
        *   Isolated – no need to consider dependencies unnecessary for that mode.
        Cons:
        *   Cannot print it ahead to analyze - need to run some sort of dry mode (which goes through to construct the nodes, but does not execute them - which is also tricky as you need to execute nodes to evaluate modes).

        How growing DAG is different from color-coding?

        Color-coded is constructed fully ahead (static). Growing is dynamic (depends on the execution of previous steps).

        Why do we need that complexity for `protoprimer`?

        It might be elegance and clarity even if it is a bit more moving parts (complexity).

## Approach

*   All state names are stable and known at the start.

    They are mostly defined by `EnvState` but can be append-only extended at start.

    Also, they are stable within current `StateStride`
    (each restart may potentially extend the set of state names).

    See the 1-dimentional approach above.

*   DAG can be built progressively (several targets).

    For example, a target that finalizes config may be used to build other parts of DAG and
    set a new target to evaluate.

*   Dependencies between state names are built progressively.

    Each state is associated with a factory which recursively invokes factories of all states it depends on.

    Can we replace that factory?
    We can use append-only extension to state names,
    but can we replace association of an existing state with a factory?

    If yes, it is wise not to define static association via `EnvState` enum in the first place
    (as it looks very confusing - you can navigate from state name to implementation, but that is not guaranteed).

    The approach is not to redefine the factory.
