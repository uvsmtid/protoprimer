## Intro

`protoprimer` may need to install dependencies from private artifact repos.

This also requires authn/authz settings for those artifact repos.

There is a choice to rely on `python` (or `pip`, or `uv`, or ...) features to disvover
system or user configuration or make it configurable for that repo clone only.

Without taking away the choice, the most autonomous approach (isolated to that repo clone only) is required.

Isolation should allow hooks that configure access to these private artifact repos during bootstrap.

## `venv` restrictions

There are several things to take care of:
1.  private artifact repo URL
2.  private artifact repo authn/authz

The 1st one (URL) is easy -
it can be external (system or user) configuration or internal (via CLI arg specified by the wrapper).

The 2nd one (authn/authz) is harder -
it often requires additional packages in `venv` (e.g. for `pip`).
To avoid relying on system or user configs,
those extra packages will have to be installed in the `venv`
before installing packages from private artifacts repos behind authn/authz.

## Test private repo scenario

This is a sample test scenario to use private repo.

```
./cmd/gcloud_artifact_repo create test-python-repo
```

```
./cmd/publish_package protoprimer --repository_url https://us-central1-python.pkg.dev/protoprimer/test-python-repo/
```

See the content of artifact repo:
https://console.cloud.google.com/artifacts/python/protoprimer/us-central1/test-python-repo?project=protoprimer

Set up new test `venv`:

```
python -m venv test-venv
test-venv/bin/pip install keyring keyrings.google-artifactregistry-auth
```

NOTE: `simple` suffix in the URL:

```
test-venv/bin/pip install protoprimer --index-url https://us-central1-python.pkg.dev/protoprimer/test-python-repo/simple
```

```
./cmd/gcloud_artifact_repo delete test-python-repo
```
